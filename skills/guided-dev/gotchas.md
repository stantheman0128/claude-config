# Gotchas — Guided Development 常見問題

## 致命級

### 1. 沒有 Verify 就進下一個 Slice
- **問題**：Build 完覺得「應該對了」就往下走，錯誤累積到最後才爆
- **正確做法**：每個 Slice 必須有外部可觀察的驗證。curl 回 200、test pass、螢幕上看到結果。「我覺得對了」不算

### 2. 問使用者技術決策
- **問題**：問初學者「要用 SQLite 還是 PostgreSQL」，他們不知道差別
- **正確做法**：自己決定，用一句話解釋。「我用 SQLite 因為不需要另外裝資料庫，最簡單。」

### 3. Spar phase 走過場
- **問題**：自我挑戰只寫「設計看起來合理」，沒有真的找到問題
- **正確做法**：至少找出 1 個潛在問題。如果真的找不到，說明為什麼這個設計特別堅固

## 重大級

### 4. Slice 切太大
- **問題**：一個 Slice 包含 server + database + API + frontend，驗證時不知道哪裡壞了
- **正確做法**：每個 Slice 2-5 分鐘工作量。一個 Slice 只做一件事

### 5. Fix 超過 3 次不 re-plan
- **問題**：同一個問題反覆修、反覆失敗，越改越亂
- **正確做法**：Fix 3 次仍失敗 → 停下來。可能是 Slice 切分有問題、設計有根本缺陷、或環境有意外限制

### 6. 修復影響已完成的 Slice 但沒有回測
- **問題**：修 Slice 3 的時候改了 Slice 1 的檔案，沒有重新驗證 Slice 1
- **正確做法**：如果改動碰到已完成 Slice 的檔案 → 重新跑那個 Slice 的驗證

### 7. 一次寫太多程式碼才 build
- **問題**：寫了 200 行才試著編譯，結果 30 個 error
- **正確做法**：寫一個邏輯單元就 build 一次。快速回饋比一次寫對更重要

## 中度級

### 8. 研究階段就開始寫程式碼
- **問題**：Phase 1 Research 就忍不住開始改檔案
- **正確做法**：Phase 1 只讀不寫。可以考慮啟用 /research-mode

### 9. 沒有用 TaskCreate 追蹤 Slice
- **問題**：做到一半忘記哪些 Slice 完成了、哪些還沒
- **正確做法**：每個 Slice 建立一個 Task，開始時 in_progress，驗證通過後 completed

### 10. 跳過 Phase 0 規模判斷
- **問題**：一個 2 行改動也走完整流程，浪費時間
- **正確做法**：Trivial 任務直接做。只有 3+ 步驟才走流程
